# DataIQ — Agent Development & Architecture Guide

Welcome! If you are an AI assistant (like Claude, Cursor, or ChatGPT) or a developer building onto or maintaining **DataIQ**, this document is your guide. It explains the system architecture, state management, data contracts, and coding conventions used throughout this repository.

---

## 1. System Philosophy & Architecture

DataIQ enforces a **clean separation of concerns** between presentation (Streamlit frontend) and business logic (modules/utils).

```
 ┌────────────────────────────────────────────────────────┐
 │                      app.py (Entry)                    │
 └───────────┬────────────────────────────────────────────┘
             │ (Routes to pages)
             ▼
 ┌────────────────────────────────────────────────────────┐
 │                   pages/*.py (View)                    │
 ├────────────────────────────────────────────────────────┤
 │ - Render layouts & interactive elements                │
 │ - Read/Write to st.session_state                       │
 │ - Invoke Chart Factory & Formatters                    │
 └───────────┬────────────────────────────────────────────┘
             │ (Computes & queries)
             ▼
 ┌───────────────────────────────┬────────────────────────┐
 │      modules/*.py (Logic)     │   utils/*.py (Helpers) │
 ├───────────────────────────────┼────────────────────────┤
 │ - Raw data processing         │ - Chart Factory        │
 │ - Machine learning checks     │ - DataType Detector    │
 │ - Groq API calling            │ - Formatting helpers   │
 └───────────────────────────────┴────────────────────────┘
```

*   **No computation in views:** Streamlit page scripts (in `pages/`) should only render UI components, handle user interactions, and fetch charts or values from computed data. All heavy lifting occurs in `modules/`.
*   **Immutable Raw Data:** `st.session_state["df_raw"]` represents the original uploaded dataset and should never be modified. All cleaning, transformation, and preprocessing output goes to `st.session_state["df_clean"]`.
*   **Error Isolation:** Every calculation module must be defensive. If an exception occurs, catch it, log it (or display a warning), and return a safe, default structure (like an empty dictionary `{}`) rather than letting the whole page crash.

## 2. Session State Blueprint

Streamlit rerun behavior re-executes the active page script from top to bottom. State persistence is managed through `st.session_state`. Below is the complete state registry:

| Key Name | Data Type | Description |
| :--- | :--- | :--- |
| `df_raw` | `pd.DataFrame` | The raw, un-cleaned dataset loaded from file. |
| `df_clean` | `pd.DataFrame` | The cleaned dataset produced by the Preprocessor module. |
| `file_name` | `str` | Name of the active file (e.g., `"titanic_sample.csv"`). |
| `file_size_mb` | `float` | File size in megabytes. |
| `profile` | `dict` | Master dictionary containing results from all analysis modules. |
| `target_col` | `str` or `None` | The selected machine learning target column (or auto-detected). |
| `problem_type` | `str` or `None` | ML type: `"classification"` (binary/multiclass), `"regression"`, `"clustering"`, or `"unknown"`. |
| `groq_report` | `str` or `None` | Cached markdown string containing the generated Groq AI report. |
| `health_score` | `dict` | Details of the dataset health score (computed by `modules.health_score`). |
| `analysis_complete`| `bool` | Flag set to `True` once the initial full analysis has run. |
| `demo_mode` | `bool` | Flag set to `True` if using a built-in sample dataset. |
| `groq_api_key` | `str` | The user's active Groq API Key. |
| `cleaning_choices` | `dict` | Saved user choices for missing values, scaling, encoding, etc. |

## 3. The `profile` Schema & Data Contracts

The `st.session_state["profile"]` dictionary is the central data exchange medium. Modules add keys to this dictionary, which the pages then consume.

### Keys inside `profile`:
*   `meta`: `{ "file_name": str, "rows": int, "columns": int, "total_cells": int, "memory_mb": float }`
*   `columns`: A dictionary mapping `column_name -> details` generated by `modules/profiler.py`.
    ```json
    {
      "Age": {
        "dtype": "Float64",
        "dtype_class": "numeric",
        "null_count": 177,
        "null_pct": 19.87
      }
    }
    ```
*   `quality`: Generated by `modules/quality.py`. Contains `missing_values` percentages, `duplicate_rows`, and `type_inconsistencies`.
*   `statistics`: Summary metrics created by `modules/statistics.py` (mean, median, std, skewness, kurtosis, etc., or frequency counts for categorical).
*   `outliers`: Outlier counts, percentages, and indices created by `modules/outliers.py`.
*   `correlations`: Multicollinearity metrics and correlation matrices (Pearson, Spearman) computed by `modules/correlations.py`.
*   `ml_readiness`: Assessment of features, class imbalance, recommended models, and potential blockers created by `modules/ml_advisor.py`.

## 4. Groq LLM & Fallback Integration

To interact with the Groq API securely and reliably, the module `modules/groq_client.py` implements a multi-model fallback list.

### Context Optimization
Because LLM context windows are limited and larger datasets cannot be sent directly, we use `build_profile_summary(profile, health_score)` to condense the `profile` object into a lightweight, highly dense JSON summary of less than 3,000 characters before sending it to the API.

### Fallback Implementation Pattern
If you write or edit functions calling the Groq API, wrap the call in a retry loop using the established fallback models:

```python
models = [
    "llama-3.3-70b-versatile",
    "llama-3.1-8b-instant",
    "qwen/qwen3-32b",
    "groq/compound"
]

last_error = None
for model in models:
    try:
        response = client.chat.completions.create(
            model=model,
            messages=...,
            # configuration options
        )
        # return the successful result (optionally prepending the model name)
        return f"> *Generated by {model}*\n\n{response.choices[0].message.content}"
    except Exception as e:
        last_error = e
        continue
```

## 5. Development & Coding Rules

When updating or extending the codebase, adhere to the following rules:

### A. Pandas 3.x Compatibility
The project is configured to run on newer pandas versions. Avoid deprecated pandas syntax:
*   **Do not use** `df.select_dtypes(include='object')` to catch only strings. Use `pd.api.types.is_string_dtype(df[col])` or `str(df[col].dtype) == "string"` to be explicit.
*   **Do not modify** dataframes in-place unless intended; use `.copy()` when manipulating slices of dataframes (especially when dealing with Numpy arrays to avoid read-only array exceptions).
*   **Time frequencies:** Use lowercase letters (e.g. `'h'` instead of `'H'`, `'d'` instead of `'D'`) for time offsets and frequencies.

### B. UI & Charting
*   Never generate plots directly inside `pages/` using raw matplotlib or plotly commands if they can be unified.
*   All charts should be added to `utils/chart_factory.py` to ensure unified styles, appropriate color scales, and dark-theme configurations.
*   Always enable responsive layout sizing in Plotly configs (`responsive=True`).

### C. Testing
*   Write unit tests under `tests/` for any new logic added in `modules/`.
*   Test files must start with `test_` so that they are picked up automatically by `pytest`.
*   Run tests locally using: `conda run -n cv_conda python -m pytest tests/ -v`.

## 6. How to Extend DataIQ

### Adding a New Analysis Module
1.  Create the file under `modules/your_module.py`.
2.  Implement a main calculation function, e.g., `def analyze_feature_X(df, columns_profile) -> dict:`. Wrap the core logic in a `try-except` block to ensure safety.
3.  Import your function in `app.py` inside the `run_full_analysis` function.
4.  Store the output in the `profile` dictionary:
    ```python
    profile["feature_x"] = analyze_feature_X(df, profile["columns"])
    ```
5.  In the corresponding `pages/` script, retrieve the data using `st.session_state["profile"].get("feature_x")` and display the results.
